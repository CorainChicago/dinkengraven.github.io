Git Cheat Sheet

A quick guide so that you don't have to ugly-cry.

Hey, we've all been there. You're plugging away at your code, happy as a clam, and suddenly realize that you forgot to switch into a feature branch before you got started. Or maybe you tried to commit changes to a file that you've already changed, and now you have merge conflicts to sort out. Or perhaps someone suggested that you simply pull your changes from your remote repository before proceeding, and you had to ask them to repeat those instructions in English. Simply use this handy reference to get yourself back on track. I made this cheat sheet after several panic-stricken moments of thinking that I just deleted my entire repo, or that I had otherwise irrevocably Ruined Everything Forever. The bad news is that it simply takes practice. The good news is that git is generally pretty forgiving.

Getting & Creating Projects

git init

Use this command to create a repository from an existing directory of files. This can be done at any time in any local directory. 


				$ pwd
				>  /Users/Documents/js
				$ git init
				> Initialized empty Git repository in /Users/Documents/js/.git/

git clone

This command will clone a repository and provide you with a local copy. This allows you to read and edit your own copies of the files without directly affecting the originals. 


				$ pwd
				>  /Users/Documents/new-project
				$ git clone https://github.com/username/my-awesome-project.git
				> Cloning into 'my-awesome-project'...
				> remote: Counting objects: 56, done.
				> remote: Total 56 (delta 0), reused 0 (delta 0), pack-reused 56
				> Unpacking objects: 100% (56/56), done.
				> Checking connectivity... done.

Branching, Fetching, Merging, and Pulling

git branch (branch name)

Use this command to create a new branch. Branching with git allows you to work on a copy of your file without making permanent changes.

$ git branch new-feature

Now you can check which branches you've made as well as your current branch (marked with an asterisk *): 


				$ git branch
				>  master
				>* new-feature

git checkout (branchname)

Use this command to switch branches. It can also be shortened to git co. 


				$ git checkout master 
				> Switched to branch 'master'
				$ git co new-feature
				> Switched to branch 'new-feature'

git fetch

Used to retrieve changes from a remote repository to update your local repo. Note that this command only retrieves the changes; the second step is to merge them.

git merge

Used to merge remote changes into your local repo.

git pull

Combines the functions of git fetch and git merge. Note that if there are no merge conflicts, this process will happen all at once with no chance to make any decisions about what to merge. If you want more options, stick with the fetch/merge process.

Remotes

Remotes are a way to explicity tell git where you want to send and receive changes. If you are working on a forked and cloned repository, you may want to be able to fetch changes from the original repo to your local copy, and push changes from your local copy to either your remote repo or the original.

git remote -v

Lists current remotes.

git remote add (remote name) (remote URL)

Create a new remote with the specified name and URL. Common names include origin (for your own remote repo) and upstream (the original repo that you forked).

git remote rm (remote name)

Removes the named remote.

git remote rename (old name) (new name)

Changes the name of an existing remote.

git remote set-url (--push, --fetch) (name) (new URL)

Change the URL associate with a remote. Use --push to specifically change the URL assocuate with push, or --fetch to change the URL associate with fetch.

Who Gives a Hoot About Version Control?

You do! Version control allows you to edit files without fear of permanently Ruining Everything Forever. If you want to add a new feature to your application, using version control lets you do that without risking breaking the entire program altogether. If you merge your new features into your existing program and something breaks, you can simply revert back to what your code looked like before you merged those changes. Then you can work on getting the bugs out of your new feature without major downtime.

Staging & Committing Changes

git status

Reviews files that have been added and changed.

git add

Used to stage files for commit. Files can be added individually, or you can use the command git add . to add all changed files in the pwd.

git commit -m "brief summary of changes"

This command saves a "snapshot" of your file in its current state. It's advisable to make multiple commits as you work on a file, so that you have the choice to return to a previous state should you need to. Be sure to include a message describing what you changed (but keep it brief).

git log

Use this command to review previous commits.

git diff

Provides a detailed log of changes made to unstaged files. Can be viewed with pagination on the command line.

Using git diff --cached will show changes made to files that have been staged but not yet committed.

git stash

Temporarily store changes. This is a great tool when you forget to switch to a feature branch before you begin working. Use git stash pop to retrieve your stashed changes after you switch branches. 


				$ git stash
				> Saved working directory and index state WIP on new-feature:
				>  d26bd39 add links
				> HEAD is now at d26bd39 add links

				$ git stash pop
				> (This will now show your git status with files to be staged)

git push

Used to push local commits to your remote repository.

Resolving Merge Conflicts

The most common type of merge conflict, an edit collision, occurs when the same file is changed on two different branches, and then you attempt to merge those branches together. If you're trying to complete a pull request on GitHub, the site will suggest that you first resolve these conflicts using the command line.

Begin by opening the file in your text editor. Git will helpfully point out the conflicts using <<<<<<< at the beginning of the problem area and >>>>>>> at the end. In between, you'll find ======= delineating the point between the two branches in conflict.

Here you can choose to keep the changes in the top half (at the HEAD), or the bottom half (the other branch). Delete whichever code you don't want to keep as well as the markers <<<<<<<. =======, and >>>>>>>. Alternatively, you can keep a combination of the two changes.

Save the file and close your text editor. You should now be able to use git add to add your changes for commit now that the conflict has been resolved.

For more information on conflicts, check out this page at help.github.com.
